# 客服项目 - 面试 Q&A 准备清单

## 一、项目层面

### Q1: 介绍一下你的项目？

**A:** 我做的是一个智能电商客服系统，核心思路是把大模型和知识库结合起来，让 AI 基于真实的商品、活动、FAQ 数据来回答客户问题，而不是凭空编造。系统采用了双引擎架构——LLM 可用时走大模型生成回复，LLM 不可用时自动降级为关键词规则引擎，保证系统永远可用。前端是 Vue 3，后端是 Spring Boot 3，通过 OneRouter 统一网关对接 LLM。

### Q2: 为什么选择做智能客服项目？

**A:** 电商平台每天面对大量重复性咨询，比如商品优惠、退换货、物流查询这些。传统做法要么纯人工成本太高，要么硬编码规则体验差。我想探索怎么用大模型来解决这个问题，同时又不能完全依赖大模型——因为 LLM API 可能不稳定，所以设计了双引擎降级机制，这在实际生产中是很有价值的。

### Q3: 项目的核心功能有哪些？

**A:** 三大模块。对话中心是核心，用户发消息后系统经过 8 步流水线处理：场景识别、意图分析、情绪判断、知识检索、多轮上下文构建、LLM 回复生成，最后把推理过程完整展示给用户。知识中心管理 5 类知识数据——商品、活动、FAQ、行业知识、解决方案。场景配置管理意图规则、提示词模板和业务规则。

### Q4: 意图识别是怎么做的？

**A:** 我用了双层架构。主路径是调用 LLM，发一个结构化的 Prompt，让大模型同时返回场景类型、意图编码、置信度和情绪，temperature 设为 0.1 保证输出确定性。如果 LLM 调用失败，就走关键词兜底——我写了三个 fallback 方法，通过 `containsAny()` 匹配预定义关键词来识别场景、意图和情绪。比如包含"优惠、打折、促销"就识别为售前咨询的商品优惠查询意图。

### Q5: 知识库检索是怎么实现的？

**A:** 检索分 5 步，有优先级。首先看前端有没有手动选择商品或活动，选了的直接用，score 给 0.99。没有手动选择就走自动关键词匹配——遍历所有商品，用 `calculateMatchScore()` 计算匹配分数，取 Top 3。这个算法先做单字符匹配，再做 2-6 字符的连续子串匹配，子串权重更高。比如搜"运动鞋"，连续命中"运动鞋"三个字比分别命中"运""鞋"得分高。最后归一化到 0-1，阈值 0.3 以上才算命中。

### Q6: 多轮对话是怎么处理的？上下文怎么管理？

**A:** 每次生成回复前，我从数据库取同 session 最近 10 条历史消息，按时间顺序排列，然后注入到 LLM 的 messages 数组中。最终发给 LLM 的消息结构是 `[system提示词, ...历史消息, 当前用户消息]`，这是 OpenAI Chat Completions API 的标准多轮格式。历史消息存在 MySQL 的 `cs_chat_message` 表里，通过 sessionId 关联，这样天然支持多会话隔离。

### Q7: LLM 是怎么集成的？调用流程是怎样的？

**A:** 我通过 OneRouter 统一网关来调用 LLM，它兼容 OpenAI 的 Chat Completions API。一次对话中 LLM 被调用两次：第一次 `analyzeUserInput()` 做意图分析，temperature 0.1、max_tokens 150，要求返回纯 JSON；第二次 `generateReply()` 生成回复，temperature 0.7、max_tokens 500，把场景、意图、情绪、检索到的知识都注入到系统提示词中。所有 LLM 调用都 try-catch 返回 null，调用方通过 null 判断走规则引擎兜底。

### Q8: 消息发送到收到回复，整个链路是怎样的？

**A:** 8 步流水线。第 1 步保存用户消息到数据库；第 2 步智能分析——LLM 分析场景、意图、情绪，失败走关键词兜底；第 3 步知识检索——按优先级查商品、活动、FAQ；第 4 步构建多轮上下文 + 提示词组装 + LLM 生成回复，失败走规则引擎；第 5 步根据意图选择卡片类型（优惠查询→活动卡片）；第 6 步保存 AI 回复到数据库；第 7 步构建 6 步推理过程供前端展示；第 8 步组装完整响应返回。整个过程用 `@Transactional` 保证事务性。

### Q9: 项目的整体架构是怎样的？

**A:** 前后端分离，前端 Vue 3 + Element Plus + Vite，后端 Spring Boot 3.4.5 + JPA + MySQL。后端分三层：Controller 接收请求，Service 处理业务逻辑，Repository 操作数据库。业务上分三个模块：Chat 模块处理对话核心链路，Knowledge 模块管理 5 类知识数据，Scene 模块管理意图、提示词和规则配置。LLM 通过 OneRouter 统一网关接入，用 RestTemplate 发 HTTP 请求。

### Q10: 数据库设计了哪些表？为什么这样设计？

**A:** 10 张表，分三组。对话相关 2 张：`cs_chat_session` 会话表和 `cs_chat_message` 消息表，一对多关系。知识相关 5 张：商品、活动、FAQ、行业知识、解决方案，每类知识字段差异大，比如商品有价格和规格，FAQ 有问答对，所以独立设计而不做通用模型，方便查询和索引优化。场景相关 3 张：意图、提示词、规则。每张知识表都预留了 `vectorized` 字段，为后续接入向量数据库做准备。

### Q11: 知识中心的五类知识是怎么组织的？

**A:** 五类知识各自独立的表和实体类，通过 `KnowledgeService` 提供统一的 CRUD 入口。其中商品、活动、FAQ 三类真正参与了对话链路——对话时会被 `ChatService.retrieveKnowledge()` 检索并注入到 LLM 提示词中。行业知识和解决方案目前是模拟状态，CRUD 可用但没接入对话链路，预留了 RAG 向量检索和流程编排的扩展方向。前端 Tab 标签上有 tooltip 角标，清楚标注了每类知识的接入状态。

### Q12: 前端四栏布局是怎么设计的？

**A:** 对话中心采用四栏布局。左侧是常见问题侧边栏（`QuestionSidebar.vue`），点击直接发送；中间是 iPhone 风格的聊天界面（`ChatInterface.vue`），支持多会话切换；右上角是知识选择区（`KnowledgeSelect.vue`），可以手动选择商品和活动作为对话上下文；右下角是 Agent 工作区（`AgentWorkArea.vue`），实时展示 6 步推理链：输入解析→场景识别→意图分析→情绪判断→知识检索→生成回复。

### Q13: 双引擎降级具体怎么实现的？

**A:** 分两个降级点。第一个在分析阶段：`LlmService.analyzeUserInput()` 调 LLM 解析意图，返回 null 时 `ChatService` 自动切到三个 fallback 方法——`recognizeSceneFallback()`、`analyzeIntentFallback()`、`analyzeEmotionFallback()`，通过预定义关键词做匹配。第二个在回复阶段：`LlmService.generateReply()` 调 LLM 生成回复，返回 null 时切到规则引擎 `generateReply()`，基于意图和知识用模板拼接回复。日志里会标注 `分析引擎=LLM` 还是 `关键词`，`回复引擎=LLM` 还是 `规则`。

### Q14: 意图识别准确率不高怎么办？

**A:** 几个优化方向。一是 Prompt 优化，当前给 LLM 的分析提示词可以加入 few-shot 示例，提高结构化输出的准确性。二是动态意图加载，目前意图是代码硬编码的 5 种，数据库里的 `cs_scene_intent` 表已经预留了，可以改成运行时从表里读取，支持动态新增意图和触发关键词。三是引入 Embedding 做语义匹配，替代当前的关键词子串匹配。四是加入置信度校验，低于阈值时要求用户澄清。

### Q15: 如果要接入微信、抖音等渠道，架构怎么设计？

**A:** 目前渠道信息已经有预留——`cs_chat_session` 表有 `channel` 字段（web/app/wechat）。扩展的话，我会加一个 Channel Adapter 层，每个渠道实现一个适配器，负责消息格式转换和接收推送。核心的 `ChatService` 对话链路不需要改，因为它只关心 sessionId 和消息内容。还需要一个消息队列（如 RabbitMQ）做异步解耦，渠道适配器接收消息后投递到队列，ChatService 消费处理。

### Q16: 高并发场景下怎么保证性能？

**A:** 当前系统有几个性能保障点：Druid 连接池管理数据库连接，Redis + Redisson 做缓存和分布式锁。如果要支撑高并发，首先 LLM 调用是最大瓶颈，可以引入请求队列和限流。知识检索目前是全表扫描，数据量大时需要加全文索引或接入 Elasticsearch。历史消息查询可以加 Redis 缓存。对话链路可以拆分成异步步骤——先快速返回"正在思考"，用 SSE/WebSocket 推送最终结果。

### Q17: 项目有什么不足？如果继续做会怎么改进？

**A:** 几个主要不足。知识检索用的是关键词子串匹配，准确率有限，应该引入向量数据库做语义检索，数据库字段已经预留了 `vectorized`。场景配置的意图、提示词、规则还没接入对话链路，应该实现从数据库动态加载。LLM 回复是同步等待的，应该改成流式输出（SSE）做打字机效果。还缺少 Token 用量统计和对话质量评估体系。如果继续做，优先级是：向量检索 > 流式输出 > 提示词动态化 > 多渠道接入。

## 二、大模型应用层面

### Q1: 大模型（LLM）是什么？有什么特点？

**A:** 大模型就是参数量非常大的预训练语言模型，比如 GPT-4、Claude 这些。它的特点是能理解自然语言语义，生成流畅的文本回复，还能做推理、分类、提取结构化信息等任务。在我的项目中，我用了 gpt-4o-mini 模型，一次对话调用两次——第一次让它分析用户意图（返回 JSON），第二次让它生成自然语言回复。核心优势是通用性强，一个模型就能处理各种客服场景。

### Q2: Prompt 是什么？怎么写好一个 Prompt？

**A:** Prompt 就是给大模型的指令文本，决定了模型的行为和输出质量。写好 Prompt 有几个要点：明确角色（"你是电商客服系统的智能分析引擎"）、明确输出格式（要求返回纯 JSON）、给出约束条件（"禁止编造信息"）、提供上下文（注入检索到的知识）。在我的项目中，分析用的 Prompt 要求模型输出 5 个维度的 JSON，生成回复的 Prompt 包含 6 条行为规则加上场景、意图、情绪、知识等上下文。

### Q3: 什么是 System Prompt 和 User Prompt？

**A:** System Prompt 是设定模型角色和行为规则的，User Prompt 是用户的具体问题。在 OpenAI 的 Chat Completions API 中，messages 数组里 role 为 system 的是 System Prompt，role 为 user 的是 User Prompt。在我的项目中，System Prompt 包含了角色设定（"你是专业电商客服"）、6 条行为规则、当前场景类型、用户意图、情绪以及检索到的知识，通过 `buildEnhancedPrompt()` 方法动态组装。

### Q4: Temperature 参数是什么意思？

**A:** Temperature 控制模型输出的随机性，范围 0-2。值越低输出越确定性，越高越有创造性。在我的项目中，意图分析用 temperature=0.1，因为需要确定性的结构化输出（JSON）；生成回复用 temperature=0.7，因为需要自然、多样化的表达。如果设为 0 就完全确定性输出，每次一样；设为 1.5 就会比较天马行空，可能跑题。

### Q5: 传统客服和 AI 客服有什么区别？

**A:** 传统客服要么纯人工（成本高、效率低），要么用硬编码的规则引擎（只能处理固定模式的问题，体验差）。AI 客服用大模型理解自然语言语义，能处理各种灵活表达，比如"最近有啥划算的"和"有什么优惠"它都能理解为同一个意图。但 AI 客服也有挑战——大模型可能"幻觉"编造信息，所以我的项目用知识库来约束，让 AI 基于真实数据回答。而且加了双引擎降级，保证 LLM 挂了系统还能用。

### Q6: 意图识别有哪些实现方式？

**A:** 常见的有三种。一是基于规则/关键词匹配，简单直接但不够灵活，我的项目里 fallback 方法就是这种。二是基于机器学习分类器，比如训练一个 BERT 模型做意图分类，准确率高但需要标注数据。三是直接用大模型，我的主路径就是让 gpt-4o-mini 分析意图，优点是零标注数据、理解能力强，缺点是调用成本高、延迟大。实际生产中通常是混合方案——简单场景走规则，复杂场景走模型。

### Q7: 什么是槽位填充（Slot Filling）？

**A:** 槽位填充是对话系统中的概念，指从用户的话里提取关键信息来填充预定义的参数。比如用户说"我想退昨天买的那双运动鞋"，需要填充的槽位有：操作=退货、商品=运动鞋、时间=昨天。我的项目目前没有显式的槽位填充模块，但 LLM 分析时实际上在做类似的事——从用户输入中提取场景、意图、情绪等结构化信息。如果要做更复杂的对话流程（比如下单），就需要正式的槽位管理。

### Q8: 多轮对话的难点在哪里？

**A:** 主要有三个难点。第一是上下文管理——历史消息越多，传给 LLM 的 Token 越多，成本和延迟都会增加，所以我限制了最近 10 条。第二是指代消解——用户说"它多少钱"，"它"指的是前面聊的哪个商品？大模型在一定程度上能处理，但不完美。第三是话题切换——用户可能突然从问优惠变成问退货，系统需要识别话题转换并重新分析意图。我的项目通过每轮都重新做意图分析来处理话题切换。

### Q9: 大模型应用和传统应用开发有什么区别？

**A:** 最大区别是不确定性。传统应用逻辑是确定的，同样的输入一定得到同样的输出。大模型的输出有随机性，而且可能出错（幻觉）。所以大模型应用需要额外考虑：输出的校验和兜底机制（比如我的双引擎设计）、Prompt 工程（怎么写提示词让模型行为可控）、成本控制（每次调用都有 Token 费用）、延迟优化（LLM 调用比数据库查询慢得多）。本质上是把"编程"变成了"编程+Prompt 设计"。

### Q10: 你觉得 AI 客服的难点在哪里？

**A:** 我觉得有三个层面。技术层面，最大难点是知识准确性——怎么保证 AI 回答的是真实信息而不是编造的，我的方案是用知识库检索来约束 LLM 输出。产品层面，难点是处理边界情况——用户的表达千变万化，总有模型理解不了的场景，需要有兜底机制和人工转接策略。工程层面，难点是稳定性——LLM API 可能超时、限流、报错，需要完善的容错设计。我的项目通过双引擎降级、全链路异常捕获来解决工程稳定性问题。

### Q11: 如何评估一个 AI 客服的效果？

**A:** 几个维度。准确率——回答是否正确、是否基于真实数据；意图识别率——是否正确理解了用户想做什么；响应延迟——用户等多久收到回复；满意度——可以在对话结束后让用户评分。在我的项目中，后台日志记录了每次对话的分析引擎（LLM还是关键词）、回复引擎、知识检索命中数、匹配分数等信息，可以做量化分析。前端推理过程的可视化也方便人工审查 AI 的决策过程。

### Q12: Token 是什么？为什么要关注 Token 数量？

**A:** Token 是大模型处理文本的最小单位，大概一个中文字约 1-2 个 Token，一个英文单词约 1 个 Token。关注 Token 有两个原因：一是成本，LLM API 按 Token 计费，输入输出都算钱；二是上下文窗口有限，比如 gpt-4o-mini 支持 128K Token，超了就截断。在我的项目中，意图分析设了 max_tokens=150（只需要短 JSON），回复生成设了 max_tokens=500，多轮历史限制最近 10 条，都是在控制 Token 消耗。

### Q13: 大模型的"幻觉"问题怎么解决？

**A:** 幻觉就是模型编造看起来合理但实际不存在的信息。常见解决方案是 RAG（检索增强生成）——先从知识库检索真实数据，再把数据注入 Prompt 让模型基于这些事实来回答。在我的项目中，就是先检索商品、活动、FAQ 数据，通过 `buildKnowledgeText()` 格式化后注入系统提示词的"参考知识"部分，并在规则中明确要求"基于提供的知识信息回答，不要编造"。这样模型的回答就会锚定在真实数据上。

### Q14: 如果让你设计一个 AI 产品，你会怎么考虑？

**A:** 首先明确用户需求和场景——AI 解决什么问题、替代了什么人工流程。然后选择技术方案——是用大模型还是传统 ML，需不需要知识库，数据从哪来。设计时重点考虑可靠性——大模型不是万能的，需要兜底方案和人工介入机制。成本也很重要——LLM 调用费用、服务器成本要在商业模式里算得过来。最后是可观测性——用户不信任黑箱，像我的项目里展示推理过程，让用户看到 AI 是怎么思考的，这对建立信任很重要。
