## 【核心1】对话流程：从用户消息到 AI 回复的 8 步链路

### 设计思路

对话链路是整个系统最核心的流程。用户发一条消息，系统需要经历 8 个步骤才能生成一条有价值的回复。

这 8 步不是随意拼凑的，每一步都有明确的输入和输出，像流水线一样串联。

### 流程全景

```
用户消息
  │
  ▼
┌──────────────────────────────────────────────────────────┐
│ 步骤1：保存用户消息                                       │
│   → ChatMessage(role=user) 写入数据库                    │
├──────────────────────────────────────────────────────────┤
│ 步骤2：智能分析（场景 + 意图 + 情绪）                      │
│   → LLM 可用：调 LlmService.analyzeUserInput()          │
│   → LLM 不可用：走关键词兜底 recognizeSceneFallback()     │
│   输出：sceneType / intent / emotion                     │
├──────────────────────────────────────────────────────────┤
│ 步骤3：知识检索                                           │
│   → 优先使用前端手动选择的商品/活动                        │
│   → 否则自动关键词匹配 Product/Activity/FAQ               │
│   输出：KnowledgeRecallVO（命中的商品、活动、FAQ）          │
├──────────────────────────────────────────────────────────┤
│ 步骤4：生成回复（含多轮上下文）                              │
│   → 从数据库取同 session 最近 10 条历史消息                 │
│   → LLM 可用：调 LlmService.generateReply()             │
│     系统提示词 + 历史消息 + 当前用户消息 → LLM              │
│     将场景/意图/情绪/知识注入增强提示词                     │
│   → LLM 不可用：规则引擎 generateReply() 兜底             │
│   输出：AI 回复文本                                       │
├──────────────────────────────────────────────────────────┤
│ 步骤5：工具选择                                           │
│   → 根据意图选择展示卡片类型                              │
│   PRODUCT_DISCOUNT → ACTIVITY_CARD                       │
│   SIZE_CONSULT → PRODUCT_CARD                            │
│   LOGISTICS_QUERY → LOGISTICS_CARD                       │
├──────────────────────────────────────────────────────────┤
│ 步骤6：保存 AI 回复                                       │
│   → ChatMessage(role=assistant) 写入数据库               │
├──────────────────────────────────────────────────────────┤
│ 步骤7：构建推理过程                                       │
│   → 组装 6 步 ReasoningStepVO 供前端展示                  │
├──────────────────────────────────────────────────────────┤
│ 步骤8：构建响应返回                                       │
│   → SendMessageVO 包含回复+卡片+推理过程                  │
└──────────────────────────────────────────────────────────┘
  │
  ▼
前端渲染：对话窗口 + Agent 工作区推理展示
```

### 关键代码

**入口方法** — `ChatService.sendMessage()`

> 文件：`backend/src/main/java/com/anjing/module/chat/ChatService.java`，第 150-246 行

```java
@Transactional
public ChatVO.SendMessageVO sendMessage(ChatDTO.SendMessageDTO dto) {
    String sessionId = dto.getSessionId();
    String userContent = dto.getContent();

    // 1. 保存用户消息
    ChatMessage userMessage = new ChatMessage();
    userMessage.setMessageId(UUID.randomUUID().toString());
    userMessage.setSessionId(sessionId);
    userMessage.setRole("user");
    userMessage.setContent(userContent);
    messageRepository.save(userMessage);

    // 2. LLM 一次性分析（场景+意图+情绪），关键词兜底
    Map<String, String> llmAnalysis = llmService.analyzeUserInput(userContent);
    if (llmAnalysis != null) {
        sceneType = llmAnalysis.get("scene");
        // ... 从 LLM 结果构建意图
    } else {
        sceneType = recognizeSceneFallback(userContent);
        intent = analyzeIntentFallback(userContent);
        emotion = analyzeEmotionFallback(userContent);
    }

    // 3. 知识检索（从数据库真实查询）
    ChatVO.KnowledgeRecallVO knowledgeRecall = retrieveKnowledge(
        userContent, intent, selectedProductIds, selectedActivityIds);

    // 4. 生成回复（优先使用LLM，回退到规则引擎）
    Map<String, Object> llmContext = new HashMap<>();
    llmContext.put("sceneType", sceneType);
    llmContext.put("intentName", intent.getIntentName());
    llmContext.put("emotion", emotion);
    llmContext.put("knowledge", buildKnowledgeText(knowledgeRecall));

    // ★ 构建多轮对话历史（取同 session 最近 10 条，排除刚保存的本轮）
    List<ChatMessage> historyMessages = messageRepository.findBySessionIdOrderByCreatedAtAsc(sessionId);
    List<Map<String, String>> chatHistory = new ArrayList<>();
    int historySize = historyMessages.size() - 1;
    int start = Math.max(0, historySize - 10);
    for (int i = start; i < historySize; i++) {
        ChatMessage m = historyMessages.get(i);
        chatHistory.add(Map.of("role", m.getRole(), "content", m.getContent()));
    }

    String llmReply = llmService.generateReply(userContent, llmContext, chatHistory);
    String aiReply = llmReply != null ? llmReply : generateReply(userContent, intent, knowledgeRecall);

    // 5. 工具选择（根据意图决定前端展示什么卡片）
    String toolChoice = selectTool(intent, knowledgeRecall);
    // PRODUCT_DISCOUNT → ACTIVITY_CARD，SIZE_CONSULT → PRODUCT_CARD，LOGISTICS_QUERY → LOGISTICS_CARD

    // 6. 保存AI回复
    ChatMessage aiMessage = new ChatMessage();
    aiMessage.setMessageId(UUID.randomUUID().toString());
    aiMessage.setSessionId(sessionId);
    aiMessage.setRole("assistant");
    aiMessage.setContent(aiReply);
    aiMessage.setCardType(toolChoice);
    messageRepository.save(aiMessage);

    // 7. 构建推理过程（前端 Agent 工作区展示的 6 步推理链）
    boolean usedLlm = llmAnalysis != null;
    List<ChatVO.ReasoningStepVO> reasoningProcess = buildReasoningProcess(
        sceneType, intent, emotion, knowledgeRecall, usedLlm);
    // 6步推理链：输入解析 → 场景识别 → 意图分析 → 情绪判断 → 知识检索 → 生成回复
    // 每步都标注了使用的分析引擎（LLM 大模型 / 关键词规则）

    // 8. 构建响应返回前端
    ChatVO.SendMessageVO response = new ChatVO.SendMessageVO();
    response.setMessageId(aiMessage.getMessageId());
    response.setContent(aiReply);
    response.setCardType(toolChoice);        // 卡片类型
    response.setSceneType(sceneType);        // 场景
    response.setIntent(intent);              // 意图
    response.setEmotion(emotion);            // 情绪
    response.setKnowledgeRecall(knowledgeRecall);   // 知识召回
    response.setReasoningProcess(reasoningProcess); // 推理过程
    return response;
}
```

### 设计要点

- **事务性**：`@Transactional` 保证用户消息和 AI 回复要么同时持久化，要么同时回滚
- **多轮上下文**：从数据库取同 session 最近 10 条历史消息，按 `system → history → user` 顺序注入 LLM，让 AI 能理解连续对话
- **日志链路**：每步都打印 `【步骤N/8】` 格式日志，包括多轮上下文条数、知识检索策略和命中详情
- **降级透明**：最终日志明确标注 `分析引擎=LLM` 还是 `关键词`，`回复引擎=LLM` 还是 `规则`

## 【核心2】意图识别：规则匹配逻辑 + 可扩展设计

### 设计思路

意图识别回答的核心问题是：**用户这句话想干什么？**

系统采用双层架构：
1. **主路径（LLM）**：让大模型理解自然语言语义，返回结构化意图
2. **兜底路径（关键词）**：预定义关键词-意图映射，覆盖最常见场景

### LLM 分析（主路径）

> 文件：`LlmService.java`，第 124-190 行

LLM 一次调用同时输出场景、意图、情绪三个维度：

```java
public Map<String, String> analyzeUserInput(String userMessage) {
    String prompt = """
        你是电商客服系统的智能分析引擎。请分析用户输入，返回纯JSON。
        分析维度：
        1. scene：场景类型（售前咨询/售后服务/物流配送/支付问题/通用咨询）
        2. intentCode：意图编码（PRODUCT_DISCOUNT/RETURN_EXCHANGE/SIZE_CONSULT/LOGISTICS_QUERY/GENERAL_QUERY）
        3. intentName：意图中文名称
        4. confidence：置信度(0-1)
        5. emotion：情绪（正面/中性/负面）
        """;
    // 调用 LLM，解析返回的 JSON
}
```

**设计要点**：
- `temperature: 0.1`：分析任务需要确定性，温度设低
- 要求返回纯 JSON，并在代码里处理 markdown 代码块包裹的情况
- 解析失败返回 null，调用方自动走关键词兜底

### 关键词兜底（降级路径）

> 文件：`ChatService.java`，第 288-321 行

三个兜底方法各自独立：

```java
// 场景识别兜底
private String recognizeSceneFallback(String content) {
    if (containsAny(content, "优惠", "活动", "打折", "促销", "价格", "多少钱")) return "售前咨询";
    if (containsAny(content, "退货", "换货", "退款", "售后", "投诉", "质量")) return "售后服务";
    if (containsAny(content, "快递", "物流", "发货", "到货", "配送")) return "物流配送";
    if (containsAny(content, "支付", "付款", "发票", "账单")) return "支付问题";
    return "通用咨询";
}

// 意图识别兜底
private ChatVO.IntentVO analyzeIntentFallback(String content) {
    if (containsAny(content, "优惠", "活动", "打折", "促销")) {
        intent.setIntentCode("PRODUCT_DISCOUNT");
        intent.setIntentName("商品优惠查询");
        intent.setConfidence(0.95);
    }
    // ... 其他意图分支
}

// 情绪识别兜底（30+ 关键词覆盖正面/负面/中性）
private String analyzeEmotionFallback(String content) {
    if (containsAny(content, "投诉", "差评", "不满", "生气", "垃圾", ...)) return "负面";
    if (containsAny(content, "感谢", "满意", "好评", "不错", ...)) return "正面";
    return "中性";
}
```

## 【核心3】知识检索：5 类知识的存储与查询

### 设计思路

客服回复的质量取决于知识库的质量。系统支持 5 类知识数据，每类有独立的存储和检索逻辑。

核心原则：**让 AI 基于真实数据回答，而不是凭空编造**。

### 知识存储

5 类知识各自一张 JPA 表，通过 `KnowledgeService` 提供统一的 CRUD。**但并非全部参与对话链路**：

```
✅ cs_knowledge_product    → 商品信息（名称、价格、描述、规格、特色） → 对话时检索注入 LLM
✅ cs_knowledge_activity   → 促销活动（活动类型、时间、规则、折扣率） → 对话时检索注入 LLM
✅ cs_knowledge_faq        → 常见问答（问题、答案、关联问题）       → 对话时关键词匹配注入 LLM
❌ cs_knowledge_industry   → 行业知识（标题、内容、来源）           → 模拟（预留 RAG 扩展）
❌ cs_knowledge_solution   → 解决方案（触发条件、处理步骤、预期效果） → 模拟（预留流程编排）
```

> 前端 Tab 标签通过 tooltip 角标和"（模拟）"标注，让使用者一目了然。5 类知识的 CRUD（新增/编辑/删除）功能全部真实可操作。

### 知识检索逻辑

> 文件：`ChatService.java`，`retrieveKnowledge()` 方法，第 335-440 行

检索按优先级分为 5 步（仅商品、活动、FAQ 三类参与，行业知识和解决方案未接入）：

```
步骤1：前端选中的商品 → 直接使用（score=0.99，source=人工选择）
步骤2：前端选中的活动 → 直接使用
步骤3：自动关键词检索商品（没有手动选择时）
   → 遍历所有商品，计算匹配分数，取 Top 3
步骤4：自动检索活动（关键词命中"优惠/活动/打折"时）
步骤5：FAQ 检索（始终执行）
   → 遍历所有 FAQ，计算匹配分数，取 Top 3
```

每步都带有详细日志，格式如：
```
  ↳ 商品检索策略: 人工选择模式（前端选中2个商品ID=[1, 3]）
  ↳ 命中商品: [1] 运动鞋Pro（人工选择，score=0.99）
  ↳ 活动检索策略: 关键词触发（命中优惠/活动等关键词）
  ↳ FAQ检索结果: 无匹配（阈值0.3）
```

> **教学要点**：讲解核心设计时，可以打开后台日志实时演示，让学生看到每一步检索的策略选择和命中详情。

### 匹配分数算法

> 文件：`ChatService.java`，`calculateMatchScore()` 方法，第 412-432 行

```java
private double calculateMatchScore(String query, String... fields) {
    // 1. 单字符匹配：查询中每个字符是否出现在目标字段中
    for (int i = 0; i < q.length(); i++) {
        if (f.indexOf(q.charAt(i)) >= 0) totalHits++;
    }
    // 2. 子串匹配（2~6字符）：连续子串命中得分更高
    for (int len = 2; len <= Math.min(q.length(), 6); len++) {
        for (int i = 0; i <= q.length() - len; i++) {
            if (f.contains(q.substring(i, i + len))) totalHits += len;
        }
    }
    // 3. 归一化到 [0, 1]，阈值 0.3 以上才命中
    return Math.min(1.0, totalHits / (double)(q.length() * 3 + totalChars * 0.5));
}
```

**算法特点**：
- 简单高效，不依赖外部搜索引擎
- 连续子串匹配权重更高（"运动鞋"命中 > "运"+"鞋"分别命中）
- 阈值 0.3 过滤低相关结果

## 【核心4】LLM 集成：如何对接 OneRouter

### 设计思路

LLM 集成的核心挑战：
1. **不同模型的 API 格式不一致** → 选择 OpenAI 兼容协议作为统一标准
2. **LLM 服务不稳定** → 所有调用都有 null 返回的降级逻辑
3. **回复质量依赖提示词** → 将上下文信息注入增强提示词

### 整体架构

```
ChatService
  │
  │ analyzeUserInput(userMessage)         — 单轮分析
  │ generateReply(userMessage, context, chatHistory)  — 多轮生成
  │
  ▼
LlmService
  │
  │ HTTP POST (OpenAI Chat Completions 格式)
  │ Authorization: Bearer {api-key}
  │ messages: [system, ...history, user]   — 多轮上下文注入
  │
  ▼
OneRouter 统一网关
  │
  │ 路由到具体模型
  ▼
gpt-4o-mini / gpt-4o / claude / ...
```

### 两次 LLM 调用

LlmService 在一次对话中被调用两次，职责完全不同：

**第一次：分析用户输入**（`analyzeUserInput`）

```
目的：理解用户意图
temperature: 0.1（需要确定性输出）
max_tokens: 150（结构化 JSON，不需要太长）
输出：{"scene":"售前咨询","intentCode":"PRODUCT_DISCOUNT","intentName":"商品优惠查询","confidence":0.95,"emotion":"中性"}
```

**第二次：生成回复**（`generateReply`）

```
目的：生成自然语言回复
temperature: 0.7（需要创造性表达）
max_tokens: 500（回复内容较长）
输入：系统提示词(含场景/意图/情绪/知识) → 历史消息(最近10条) → 当前用户消息
输出：自然语言回复文本
```

> **多轮上下文**：`chatHistory` 参数注入同 session 最近 10 条历史消息（role + content），让 LLM 理解对话上下文。传 null 则退化为单轮。

### 增强提示词构建

> 文件：`LlmService.java`，`buildEnhancedPrompt()` 方法，第 207-236 行

```java
private String buildEnhancedPrompt(Map<String, Object> context) {
    StringBuilder sb = new StringBuilder(SYSTEM_PROMPT);
    // 注入场景
    sb.append("\n当前场景：").append(context.get("sceneType"));
    // 注入意图
    sb.append("\n用户意图：").append(context.get("intentName"));
    // 注入情绪（负面情绪触发安抚策略）
    if ("负面".equals(emotion)) {
        sb.append("\n请特别注意安抚用户情绪，表达理解和歉意。");
    }
    // 注入检索到的知识
    sb.append("\n\n参考知识：\n").append(context.get("knowledge"));
    return sb.toString();
}
```

最终发送给 LLM 的系统提示词包含：
- 基础角色设定（6条规则）
- 当前场景类型
- 用户意图
- 用户情绪（负面情绪时追加安抚指令）
- 检索到的相关知识（商品/活动/FAQ）

### 容错处理

```java
// chatHistory: 同 session 的历史消息（role + content），传 null 则退化为单轮
public String generateReply(String userMessage, Map<String, Object> context,
                            List<Map<String, String>> chatHistory) {
    if (!enabled || apiKey == null || apiKey.isEmpty()) return null;

    try {
        String enhancedSystemPrompt = buildEnhancedPrompt(context);
        List<Map<String, String>> messages = new ArrayList<>();
        messages.add(Map.of("role", "system", "content", enhancedSystemPrompt));

        // ★ 注入多轮历史消息
        if (chatHistory != null && !chatHistory.isEmpty()) {
            messages.addAll(chatHistory);
        }

        messages.add(Map.of("role", "user", "content", userMessage));
        // ... 发送请求，解析响应
        return content;
    } catch (HttpClientErrorException | HttpServerErrorException e) {
        log.error("LLM 调用失败: status={}", e.getStatusCode());
        return null;
    } catch (Exception e) {
        log.error("LLM 调用异常: error={}", e.getMessage());
        return null;
    }
}
```

**关键设计**：
1. 所有异常都 catch 并返回 null，调用方通过 null 判断走规则引擎兜底
2. messages 数组按 `system → history → user` 顺序组装，这是 OpenAI Chat Completions API 的标准多轮格式

```java
String llmReply = llmService.generateReply(userContent, llmContext, chatHistory);
String aiReply = llmReply != null ? llmReply : generateReply(userContent, intent, knowledgeRecall);
```

- `enabled: true` + Key 有效 → LLM 正常工作
