# 02 - 项目架构设计

---

## 一、前后端分层架构

### 整体分层

```
┌─────────────────────────────────────────────┐
│  前端层 (Vue 3 + Element Plus + Vite)       │
│  views/ → api/ → utils/http → Axios        │
├─────────────────────────────────────────────┤
│  接入层 (Controller)                         │
│  参数校验 → 日志切面 → 统一响应              │
├─────────────────────────────────────────────┤
│  业务层 (Service)                            │
│  核心逻辑 → LLM调用 → 知识检索              │
├─────────────────────────────────────────────┤
│  数据层 (Repository + Entity)               │
│  JPA自动映射 → MySQL 10张表                  │
├─────────────────────────────────────────────┤
│  基础设施 (Config + Aspect + Util)          │
│  CORS │ 连接池 │ AOP日志 │ 异常处理         │
└─────────────────────────────────────────────┘
```

### 前端请求流程

```
用户操作
  → Vue 组件 (views/customer-service/chat/index.vue)
    → API 函数 (api/customer-service/chat.ts)
      → Axios 封装 (utils/http/index.ts)，自动加 Token、拦截错误
        → /api/* 代理到后端 localhost:10002
```

### 后端请求流程

```
HTTP 请求
  → WebConfig (CORS 放行)
    → ControllerLogAspect (记录请求日志)
      → Controller (参数接收、调用 Service)
        → Service (业务逻辑)
          → Repository (JPA 操作数据库)
    → GlobalExceptionHandler (异常统一处理)
  → R<T> 统一响应格式
```

## 二、业务模块划分

系统围绕电商客服场景，划分为三大独立模块：

```
com.anjing.module/
├── chat/           # 对话中心 —— 核心对话链路
├── knowledge/      # 知识中心 —— 5类知识数据管理
└── scene/          # 场景配置 —— 意图/提示词/规则
```

### 模块1：Chat（对话中心）

**职责**：处理用户对话的完整生命周期

| 类 | 职责 |
|----|------|
| `ChatController` | 会话 CRUD + 消息发送 API |
| `ChatService` | 核心对话链路（8步流水线）+ 多轮上下文构建 |
| `LlmService` | LLM 网关调用（分析 + 生成），支持注入历史消息实现多轮对话 |
| `ChatSession` | 会话实体 |
| `ChatMessage` | 消息实体（也作为多轮历史消息的数据来源） |

### 模块2：Knowledge（知识中心）

**职责**：管理客服所需的领域知识

| 类 | 职责 |
|----|------|
| `KnowledgeController` | 5 类知识的统一入口 |
| `KnowledgeService` | CRUD + 总览 + 导入导出 |
| `Product / Activity / Faq` | 3 个实体类 — ✅ 真实参与对话链路（被 ChatService 检索） |
| `Industry / Solution` | 2 个实体类 — ❌ 模拟（CRUD 可用，未接入对话链路） |

> 前端 Tab 标签通过 tooltip 角标清晰标注每类知识的链路接入状态。

### 模块3：Scene（场景配置）— 全部标注为（模拟）

**职责**：配置对话行为的规则和模板（CRUD 真实可操作，但尚未接入对话核心链路）

| 类 | 职责 |
|----|------|
| `SceneController` | 意图/提示词/规则的统一入口 |
| `SceneService` | CRUD + 提示词测试（真实调 LLM）+ 规则启停 |
| `Intent / Prompt / Rule` | 3 个实体类 — ❌ 全部模拟（预留动态化扩展） |

> 前端所有 Tab 均标注"（模拟）"，tooltip 详细说明当前状态和扩展方向。

### 模块间关系

```
Scene（配置）──管理──→ 意图类型、提示词模板、场景规则
     │
     │ ⚠️ 当前为（模拟）：CRUD 真实，但 ChatService 尚未从此读取
     │    预留扩展：运行时从数据库加载意图规则 / 提示词模板
     │
Chat（对话） ──检索──→ Knowledge（知识）
     │                    │
     │                    ├─ 商品/活动/FAQ → ✅ 真实参与对话
     │                    └─ 行业/解决方案 → ❌ 模拟（仅数据管理）
     │
     │ 调用（携带最近10条历史消息实现多轮上下文）
     ▼
LlmService ──请求──→ OneRouter LLM 网关
```

> **注意**：当前对话链路中的意图识别和提示词均在 `ChatService` / `LlmService` 代码中硬编码实现，Scene 模块的配置数据暂未被消费。这是有意为之的教学设计——展示「管理后台预留 + 核心链路独立」的架构模式。

## 三、数据模型设计（10 个 JPA 实体）

### 实体关系总览

```
ChatSession (会话)
  │ 1:N
  ▼
ChatMessage (消息)

Product (商品知识)     ─┐
Activity (活动知识)    ─┤
Faq (FAQ知识)          ─┤── 5类独立知识，被 ChatService 检索
Industry (行业知识)    ─┤
Solution (解决方案)    ─┘

Intent (意图配置)      ─┐
Prompt (提示词模板)    ─┤── 3类场景配置
Rule (场景规则)        ─┘
```

### 对话相关（2 张表）

**cs_chat_session（会话表）**

| 字段 | 类型 | 说明 |
|------|------|------|
| session_id | VARCHAR(64) PK | 会话唯一ID |
| user_id | VARCHAR(64) | 用户ID |
| user_name | VARCHAR(100) | 用户名 |
| channel | VARCHAR(20) | 渠道（web/app/wechat） |
| status | VARCHAR(20) | 状态（active/closed） |
| created_at | DATETIME | 创建时间 |

**cs_chat_message（消息表）**

| 字段 | 类型 | 说明 |
|------|------|------|
| message_id | VARCHAR(64) PK | 消息唯一ID |
| session_id | VARCHAR(64) | 所属会话 |
| role | VARCHAR(20) | 角色（user/assistant） |
| content | TEXT | 消息内容 |
| card_type | VARCHAR(50) | 卡片类型（PRODUCT_CARD 等） |
| created_at | DATETIME | 创建时间 |

### 知识相关（5 张表）

以商品表 `cs_knowledge_product` 为例：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| product_name | VARCHAR(200) | 商品名称 |
| product_code | VARCHAR(50) | 商品编码 |
| category | VARCHAR(50) | 分类 |
| brand | VARCHAR(100) | 品牌 |
| price | DECIMAL | 价格 |
| description | TEXT | 描述 |
| specifications | TEXT | 规格（JSON） |
| features | TEXT | 特色 |
| tags | VARCHAR(500) | 标签 |
| vectorized | BOOLEAN | 是否已向量化 |
| status | VARCHAR(20) | 状态 |

> 其他 4 张表（Activity、Faq、Industry、Solution）结构类似，每张表都有 `vectorized` 字段预留向量化扩展。

### 场景相关（3 张表）

以意图表 `cs_scene_intent` 为例：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| intent_name | VARCHAR(100) | 意图名称 |
| intent_code | VARCHAR(50) | 意图编码 |
| scene_type | VARCHAR(50) | 所属场景 |
| trigger_keywords | TEXT | 触发关键词 |
| confidence_threshold | DOUBLE | 置信度阈值 |
| priority | INT | 优先级 |
| status | VARCHAR(20) | 状态 |

## 四、关键设计决策

### 决策1：为什么用 JPA 而不是 MyBatis？

- 项目重点是业务逻辑演示而非复杂 SQL
- JPA 的 `ddl-auto: update` 实现零 SQL 建表，降低环境搭建门槛
- 实体类即表结构，代码即文档
- 适合教学场景：改实体字段 → 重启 → 自动变更表结构

### 决策2：为什么知识表独立设计而不做通用模型？

- 每类知识字段差异大（商品有价格/规格，FAQ 有问答对）
- 独立表方便独立查询和索引优化
- 代码可读性高，每类知识对应清晰的实体类
- 预留 `vectorized` 字段，为后续接入向量数据库做准备

### 决策3：为什么 LLM 做双引擎降级？

- LLM API 不稳定（网络、限流、Key 过期），线上不能挂
- 关键词规则引擎作为兜底，保证基础功能始终可用
- 日志明确标注「分析引擎=LLM」还是「关键词兜底」，方便排查

### 决策4：前端为什么用四栏布局？

- 客服场景需要同时展示：问题列表、对话窗口、知识上下文、推理过程
- 四栏布局让信息一目了然，适合教学演示和实际运营

## 六、本节小结

通过本节，学生应该理解：
1. 请求从浏览器到数据库的完整链路
2. 三大业务模块各自的边界和协作方式
3. 10 个数据表的设计意图和字段含义
4. 每个设计决策背后的原因（不是"恰好这样做"，而是"有理由这样做"）
5. 代码目录的组织方式，能快速定位任何功能的实现位置
